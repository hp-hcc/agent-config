## 全局规则增补

- 后续执行任务时，如判断某条规则适合沉淀到全局 `AGENTS.md`，不要直接修改。
- 提议时必须明确说明：为什么适合放全局、它解决的是什么重复问题、为什么不适合放到父目录 `AGENTS.md`、项目级 `AGENTS.md`、文档或 skill，以及建议加入的具体文字。
- 只有在我明确同意后，才能修改全局 `AGENTS.md`。
- 每次新增全局规则时，必须同步检查全文是否存在可合并、可压缩、可收口的重复表达；如能精简，应在不丢失约束和边界的前提下同步精简全文，而不是只做追加。
- 只有当该规则属于跨项目通用规则，且已经重复暴露或明显会持续复发时，才提议加入全局 `AGENTS.md`；一次性经验、项目专属规则、技术栈专属规则不要提议进入全局。

## 工具问题识别

- `apply_patch verification failed`、`Failed to find expected lines`、补丁上下文未匹配时，默认先按补丁定位失败处理，不直接归类为工具异常。
- `apply_patch` 通道报错、Windows 沙箱刷新失败、补丁工具偶发不可用、`windows sandbox: setup refresh failed with status exit code: 1` 这类情况，默认优先按工具层异常处理。
- 无论是补丁定位失败还是工具层异常，都不要直接等同于补丁内容、业务方案、代码逻辑或需求理解有问题。

## 工具 fallback 处理

- 遇到补丁定位失败时，先回读目标文件实际内容，确认是否存在文本差异、空行差异、换行符差异或文件已被改动。
- 对短文件、结构简单文件，或 Markdown、README、说明文档、规范文档等整段调整场景，如第一次补丁失配，优先直接整段替换或整文件重写，不继续反复修补丁上下文。
- 确认属于工具层异常后，如工作区仍可写，默认直接切换到 PowerShell 写入，不在同一补丁上反复尝试。
- 修改工作区外文件时，先确认真实路径、读写权限和是否需要越权，再执行写入。
- 所有 fallback 写入完成后必须补回读校验；工作区内文件还应补 `git diff --check` 或必要的 diff 确认，工作区外文件至少完成回读校验。
